# エンハンス要望整理

更新日: 2026-04-19

目的:
- 現在上がっている改善要望を、実装判断しやすい単位へ整理する
- そのまま Issue 化しやすい形で、論点と優先度を明確にする
- 技能配分 UI の改善要望と、今後追加したいマニュアル整備を切り分ける

前提:
- 要望の中心はキャラクター作成フロー内の技能配分画面
- 初心者が迷いにくい導線と、配分判断のしやすさが主目的
- トップページのスクリーンショット付きマニュアルは別タスクとして扱う

## 要望一覧（原文ベース整理）

### A. 技能おすすめ表示・配分支援
- 職業別で絞り込みした上で、「おすすめ」順に並び替えしてほしい
- 「とりあえずこれ振っとけ」を一覧化する
- 「おすすめ」ボタンを押すだけで自動的に振り分けする
- 「おすすめ度」を「★★★☆☆」のように表示して判断しやすくしたい

### B. 技能ごとの説明強化
- 技能それぞれがどういうケースで関係するものなのか、ヘルプボタン押下時に表示してほしい

### C. 技能配分画面の見え方改善
- 右カラムの「サマリ」に、各技能の「初期値● + ●」をまとめてほしい
- 技能配分画面上で、振り分けを行った各技能の数値を確認したい

### D. 計算ルールと制約の改善
- 職業技能 + 趣味技能で上限値を振り切れる事象（計算上）の解消

### E. 職業入力・職業説明の柔軟化
- 職業に「無職」や「自宅警備員」など、自由入力可能にしてほしい
- 選択中の職業別の「平均値」が説明欄に記載あると嬉しい

### F. 今後の別タスク
- トップページに各機能と操作のマニュアルを配置したい
- マニュアルは機能画面のスクリーンショットを含めつつ説明を付与したい

## Spec 1. 技能配分の上限ロジック統一
対応候補:
- I-8 技能配分の上限ロジック統一

対象画面:
- キャラクター作成ウィザードの技能配分画面
- 主に CharacterWizard.tsx の技能配分ロジック

現状の問題:
- `occupationAssigned` と `optionalAssigned` が別配列で管理されている
- `setSkillAssigned()` は更新対象の配列だけを見て clamp している
- そのため、同一 skillId に対して職業側と趣味側の両方から振った場合、片側の更新時点では `max - base` を超えなくても、合算結果として上限超過が起こりうる

仕様方針:
- 上限判定は「base + occupationAssigned + optionalAssigned <= max」で統一する
- 職業ポイント残量と趣味ポイント残量の判定は従来通り別管理とする
- 技能の最終値はどの画面でも同じ計算式で求める

データ仕様:
- `AssignedSkill` の型は当面維持する
- ただし `total` を各配列内の局所値として扱うのではなく、保存前・描画前に合算値で再計算する方針へ寄せる
- 将来的には `total` を永続値として持たず、`base` と 2 系統の配分値から導出する形も検討する

ロジック変更案:
- `setSkillAssigned(type, index, nextAssigned)` 内で、対象 skillId に対する「反対側バケットの assigned 値」を取得する
- `maxFromSkill` は `target.max - target.base - otherBucketAssigned` で算出する
- clamp は以下で行う

	- `maxFromBudget`: 現在のバケット内で残っている配分可能点
	- `maxFromSkill`: 同一技能の反対側配分も含めた上限までの残り
	- `clampedAssigned = min(nextAssigned, maxFromBudget, maxFromSkill)`

- `buildCharacterSheet()` 実行前に、全 skillId について職業側と趣味側の合算結果を再計算し、上限超過がないことを保証する
- 初期ロード時の `mergeAssignedSkills()` 後も、必要なら上限超過を補正する sanitize を 1 回通す

UI変更案:
- 技能カードの補助表示を次のように変更する
	- 現在: `初期 X / 現在 Y / 上限 Z`
	- 変更後: `初期 X / 職業 +A / 趣味 +B / 現在 Y / 上限 Z`
- 同一技能に対して反対側ですでに配分済みの場合は、その値を確認できるようにする
- これ以上振れない場合は理由を明示する
	- 上限到達
	- 職業ポイント不足
	- 趣味ポイント不足

バリデーション:
- 技能配分画面で上限超過が発生しない
- 確認画面で合計値が技能配分画面と一致する
- 保存後の再編集で値が崩れない
- CSV/XLSX 出力と再取込後も上限超過が再発しない

完了条件:
- 同一 skillId に対して職業ポイントと趣味ポイントを別々に振っても、最終値が上限を超えない
- 技能カード・サマリ・確認画面で表示される値が一致する
- 上限到達時の挙動が `+1/+5` と直接入力の両方で統一される

テスト観点:
- 上限 80 / 初期 25 の技能に対して、職業 40・趣味 20 を入れようとすると趣味側が 15 で止まる
- 趣味側で先に配分した後、職業側を増やしても合算上限を超えない
- 既存保存データの再編集時に値が意図せず減りすぎない

依存:
- なし

## Spec 2. 技能用途ヘルプの追加
対応候補:
- I-9 技能用途ヘルプの追加

対象画面:
- キャラクター作成ウィザードの技能配分画面
- 必要に応じて早引きやキャラクター参照への横展開を検討する

現状の問題:
- 技能カードには `description` が表示されているが、短い 1 行説明に留まる
- 初心者にとっては「どういう場面で振る技能か」「どの技能と使い分けるか」が分かりにくい
- 能力値や派生値には `?` ヘルプがあるのに、技能には同等の補助導線がない

仕様方針:
- 技能ごとにヘルプボタンを追加し、押下時に「用途・典型場面・初心者向け補足」を表示する
- 既存の能力値ヘルプと操作感を揃え、学習コストを増やさない
- 最小実装では技能配分画面のみ対象とする

データ仕様:
- `Skill` 型に次の任意項目を追加する
	- `usageSummary`: どういう行為に使う技能かの短文
	- `usageExamples`: 典型的な使用場面の配列
	- `beginnerHint`: 初心者向けの補足文
- 既存の `description` はカード本文の短い説明として維持する
- データ未整備の技能は以下のフォールバックで表示する
	- `usageSummary`: `description`
	- `usageExamples`: 空配列
	- `beginnerHint`: 汎用文言

UI変更案:
- 各技能カードの skill 名付近に `?` ボタンを追加する
- 押下時は技能配分画面内の共通ヘルプパネルへ詳細を表示する
- 表示内容:
	- 技能名
	- 何に使う技能か
	- 典型的な場面 2〜3 件
	- 初心者向けの見方
- モバイルでは画面下部に積まれるパネル、PC ではカード群の下または横に固定表示する

インタラクション:
- 同じ技能の `?` を再押下すると閉じる
- 別技能の `?` を押すと内容を差し替える
- 何も選ばれていないときは「技能ヘルプ」初期案内を表示する

実装上の補足:
- 能力値ヘルプで使っている `activeCharacteristicTip` と同様の状態管理を追加する
- 技能ヘルプは occupation / optional のどちらから押しても同じ skillId へ解決する
- 早引き機能の技能データとも同じマスタを参照するため、将来的な再利用を前提に型だけは共通化しておく

完了条件:
- すべての技能カードからヘルプを開ける
- 少なくとも主要技能に用途説明が入っている
- skill データに説明が未整備でも画面が壊れない
- 初心者ガイドへ戻らず、その場で技能の用途を判断できる

テスト観点:
- 職業技能・趣味技能の両一覧で `?` ボタンが表示される
- 任意の技能でヘルプを開閉できる
- データ未設定技能でもフォールバック文言が表示される
- ダークモードでも可読性が維持される

依存:
- 技能マスタの項目拡張
